3.2 Functional Requirements

This means:

What should the digital twin be able to do?

These are the functions or tasks of your system.

For your thesis, this chapter can be written like this:

What you write there

You describe the concrete capabilities your digital twin should have.

Example content

You can write things like:

The system should import and process real-world traffic data from available sources.
The system should represent the selected road segment in SUMO.
The system should integrate traffic signal control information.
The system should support calibration using measured detector data.
The system should allow validation of simulated traffic states against observed data.
The system should provide a basis for short-term traffic forecasting.
The system should be able to represent the traffic state for online and offline use cases.
In thesis style, this becomes:

You do not just list features like software documentation.

The digital twin must be capable of integrating heterogeneous traffic-related data sources, including detector data, traffic signal control information, and process data. Furthermore, it must provide a realistic microscopic representation of the selected corridor in SUMO. The system should support both offline applications, such as calibration and AI model training, and online applications, such as near-real-time traffic state estimation and short-term forecasting.

That fits your topic very well, because your assignment says the digital twin should use detector data, traffic signal control information, and process data, and should support both offline and online use cases.

3.3 Non-Functional Requirements

This means:

How should the system perform these functions?
Not what it does, but how well or under which conditions.

These are quality requirements.

What you write there

Here you describe requirements such as:

accuracy
timeliness
robustness
scalability
interpretability
data consistency
For your thesis, possible non-functional requirements are:
The system should process data with sufficient temporal consistency for real-time or near-real-time use.
The model should be accurate enough to reproduce observed traffic conditions.
The system should be robust to incomplete or noisy data.
The framework should be modular so that additional data sources can be integrated later.
The model should remain computationally feasible for the selected corridor.
The forecasting component should produce predictions within an operationally useful time frame.
In thesis style:

In addition to functional capabilities, the digital twin must satisfy several non-functional requirements. Since the system is intended for real-time or near-real-time applications, timely data processing is essential. Moreover, the model should achieve a sufficient level of accuracy in reproducing observed traffic conditions. The framework should also be robust against missing or inconsistent input data and should be designed in a modular manner to enable future extensions.

Simple way to think about 3.3:

Ask yourself:

“How good should the system be?”
or
“What quality conditions must it satisfy?”

This means:

For your topic, these components may include:

Very simple structure

You can describe the architecture as a pipeline:

Real-world data is collected
Data is cleaned and prepared
SUMO model is built and updated
Signal control data is integrated
Simulation outputs are compared with real data
Forecasting model uses the resulting data for prediction
Results are evaluated
In thesis style:

The system architecture of the proposed digital twin consists of several interconnected components. First, traffic-related data from detectors, traffic signal control, and other available operational sources are collected and preprocessed. These data are then used to construct, calibrate, and update a microscopic simulation model in SUMO. The simulation model serves as the core of the digital twin and provides traffic state information for analysis and forecasting. Based on the calibrated simulation environment and measured traffic data, a forecasting module is implemented for five-minute traffic prediction. Finally, the performance of the overall framework is evaluated using quantitative and qualitative metrics.
